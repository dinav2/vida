## 1. Overview

**vida** is a lightweight, keyboard-driven command palette for Wayland/Hyprland. It runs as a persistent background daemon with a hidden UI window that appears instantly on a global hotkey. Input is routed through a priority chain: calculator → web shortcuts → app launcher → AI provider fallback.

**Hard constraints:**
- Daemon idle RAM: < 30 MB RSS
- UI show latency: < 100 ms from hotkey to first frame
- Daemon startup: < 200 ms to IPC socket ready
- No Electron, no X11/XWayland
- No CGo in hot paths where avoidable

## 2. Functional Requirements

### FR-01 · Daemon Lifecycle
The `vida-daemon` process must start, stay resident, and recover gracefully.

| Sub-ID | Requirement |
|--------|-------------|
| FR-01a | Daemon creates Unix socket at `$XDG_RUNTIME_DIR/vida.sock` (fallback: `/run/user/$UID/vida.sock`) on startup |
| FR-01b | Daemon removes stale socket file on startup if previous process is dead |
| FR-01c | Daemon handles SIGTERM/SIGINT gracefully: close socket, flush DB writes, exit 0 |
| FR-01d | Daemon logs structured JSON to stderr (level: info/warn/error) |
| FR-01e | Daemon idle RSS must be < 30 MB after app index is built |

### FR-02 · UI Lifecycle
The `vida-ui` process manages the Wayland layer-shell window.

| Sub-ID | Requirement |
|--------|-------------|
| FR-02a | UI process starts alongside daemon and immediately hides its window (not exits) |
| FR-02b | UI window is a `zwlr-layer-shell-v1` surface, `layer: overlay`, `anchor: top`, keyboard mode `exclusive` when visible |
| FR-02c | UI window appears on the currently focused Wayland output |
| FR-02d | UI appears within 100 ms of receiving `show` IPC message |
| FR-02e | UI hides (does not exit) on Escape keypress |
| FR-02f | UI hides on focus loss (another window receives keyboard focus) |
| FR-02g | UI clears its input field and result list on hide |
| FR-02h | UI input field receives keyboard focus immediately on show |

### FR-03 · IPC Protocol
The daemon and UI communicate over the Unix socket using newline-delimited JSON.

| Sub-ID | Requirement |
|--------|-------------|
| FR-03a | All messages are single-line JSON terminated by `\n` |
| FR-03b | Each IPC connection is a persistent bidirectional stream for its session |
| FR-03c | The `vida` CLI connects, sends one command message, reads one response, then closes |
| FR-03d | The UI maintains a persistent connection to the daemon |
| FR-03e | Unknown message types must be responded to with `{"type":"error","message":"unknown type"}` rather than crashing |

**Message schema:**

```
// Client → Daemon
{ "type": "show" }
{ "type": "hide" }
{ "type": "query", "id": "<uuid>", "input": "<string>" }
{ "type": "cancel", "id": "<uuid>" }
{ "type": "launch", "app_id": "<desktop_id>" }
{ "type": "reload" }
{ "type": "clear_history" }
{ "type": "ping" }

// Daemon → Client
{ "type": "pong" }
{ "type": "show" }
{ "type": "result", "id": "<uuid>", "kind": "calc",     "value": "<string>" }
{ "type": "result", "id": "<uuid>", "kind": "shortcut", "url": "<string>", "label": "<string>" }
{ "type": "result", "id": "<uuid>", "kind": "app_list", "apps": [AppResult] }
{ "type": "result", "id": "<uuid>", "kind": "ai_token", "token": "<string>", "done": false }
{ "type": "result", "id": "<uuid>", "kind": "ai_token", "token": "",         "done": true }
{ "type": "result", "id": "<uuid>", "kind": "ai_error", "message": "<string>" }
{ "type": "error",  "message": "<string>" }
{ "type": "ok" }
```

**AppResult schema:**
```json
{
  "id": "firefox.desktop",
  "name": "Firefox Web Browser",
  "description": "Browse the World Wide Web",
  "icon": "firefox",
  "exec": "firefox %u",
  "score": 0.92
}
```

### FR-04 · Query Router
The daemon routes each query through a fixed priority chain. The first match wins.

| Priority | Handler | Match Condition |
|----------|---------|-----------------|
| 1 | Calculator | Input matches math expression pattern (see TR-04) |
| 2 | Web shortcut | Input starts with a configured shortcut prefix followed by space |
| 3 | App launcher | Input matches one or more `.desktop` entries with score > 0 |
| 4 | AI provider | Fallback — all remaining input |

| Sub-ID | Requirement |
|--------|-------------|
| FR-04a | Router evaluates handlers in priority order; first match returns its result |
| FR-04b | Calculator result is returned before any other result for matching input |
| FR-04c | App launcher results are returned concurrently with an AI result when both match (app list first, AI streams below) |
| FR-04d | Empty input returns no result and does not invoke AI |
| FR-04e | Input shorter than 2 characters does not invoke AI |

### FR-05 · Calculator
Local math evaluation with no AI call or network request.

| Sub-ID | Requirement |
|--------|-------------|
| FR-05a | Supports operators: `+`, `-`, `*`, `/`, `^` (power), `%` (modulo) |
| FR-05b | Supports functions: `sqrt()`, `abs()`, `floor()`, `ceil()`, `round()`, `log()`, `sin()`, `cos()`, `tan()` |
| FR-05c | Supports parentheses grouping |
| FR-05d | Supports integer and float literals |
| FR-05e | Result displayed inline in UI immediately (< 5 ms) |
| FR-05f | Division by zero returns error string, does not crash |
| FR-05g | Non-numeric results (e.g. `hello + world`) treated as non-match, falls through to next handler |

### FR-06 · App Launcher
Index and search installed applications via `.desktop` files.

| Sub-ID | Requirement |
|--------|-------------|
| FR-06a | Indexes `.desktop` files from: `/usr/share/applications`, `/usr/local/share/applications`, `~/.local/share/applications`, and paths in `$XDG_DATA_DIRS` |
| FR-06b | Index is built on daemon startup and rebuilt on `reload` command |
| FR-06c | Index build must not block IPC socket from accepting connections |
| FR-06d | Fuzzy search matches against: `Name`, `GenericName`, `Comment`, `Keywords` fields |
| FR-06e | Results are ranked by fuzzy score descending; up to `ui.max_results` shown |
| FR-06f | `NoDisplay=true` entries are excluded from results |
| FR-06g | Launching an app expands `%u`, `%f`, `%U`, `%F` exec placeholders and runs via `xdg-open` or direct exec |
| FR-06h | Launched app is recorded in query history |

### FR-07 · Web Search Shortcuts
Prefix-based URL expansion to open browser searches.

| Sub-ID | Requirement |
|--------|-------------|
| FR-07a | Shortcuts defined in `[search.shortcuts]` config table as `prefix = "url_template"` |
| FR-07b | URL template uses `%s` as the query placeholder |
| FR-07c | Query portion is URL-encoded before substitution |
| FR-07d | Browser opened via `xdg-open` |
| FR-07e | Default shortcuts provided: `g` (Google), `gh` (GitHub), `yt` (YouTube), `dd` (DuckDuckGo) |
| FR-07f | User-defined shortcuts in config override defaults for matching prefix |

### FR-08 · AI Provider
Pluggable AI query with streaming token output.

| Sub-ID | Requirement |
|--------|-------------|
| FR-08a | `AIProvider` interface defined; implementations are `ClaudeProvider` and `OpenAIProvider` |
| FR-08b | Active provider selected by `ai.provider` config key; changing it requires only config edit + reload |
| FR-08c | API key read from config field first, then from environment variable (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`) |
| FR-08d | Provider streams response tokens to daemon as they arrive; daemon forwards each as `ai_token` IPC message |
| FR-08e | UI renders tokens as they arrive (streaming display, not wait-for-complete) |
| FR-08f | Queries can be cancelled via `cancel` IPC message; in-flight HTTP request must be aborted |
| FR-08g | On provider error (network, auth, rate limit), daemon sends `ai_error` IPC message with human-readable message |
| FR-08h | System prompt instructs AI to give concise responses suitable for a command palette (no markdown headers, no lengthy preamble) |
| FR-08i | `ClaudeProvider` uses Anthropic Messages API with `stream: true` |
| FR-08j | `OpenAIProvider` uses OpenAI Chat Completions API with `stream: true` |

### FR-09 · History Persistence
Query history stored in SQLite, accessible via UI and CLI.

| Sub-ID | Requirement |
|--------|-------------|
| FR-09a | SQLite DB at `$XDG_DATA_HOME/vida/vida.db` (default: `~/.local/share/vida/vida.db`) |
| FR-09b | Schema: `queries(id INTEGER PRIMARY KEY, input TEXT, result_kind TEXT, result_preview TEXT, created_at INTEGER)` |
| FR-09c | Every completed query (regardless of result kind) is written to history |
| FR-09d | History capped at 500 rows; oldest are pruned on insert when over limit |
| FR-09e | Up arrow in UI input field cycles through previous inputs (most recent first) |
| FR-09f | History is accessible before app index rebuild completes |

### FR-10 · Configuration
TOML config file with sensible defaults.

| Sub-ID | Requirement |
|--------|-------------|
| FR-10a | Config file at `$XDG_CONFIG_HOME/vida/config.toml` (default: `~/.config/vida/config.toml`) |
| FR-10b | Missing config file → daemon starts with all defaults, no error |
| FR-10c | Invalid TOML → daemon logs error and exits with code 1 |
| FR-10d | `vida reload` command sends reload IPC message; daemon re-reads config and re-indexes apps |
| FR-10e | Reload does not restart daemon or UI process |

### FR-11 · CLI Commands

| Command | Behaviour |
|---------|-----------|
| `vida show` | Sends `show` to daemon; daemon signals UI to appear |
| `vida hide` | Sends `hide` to daemon; daemon signals UI to hide |
| `vida reload` | Sends `reload`; daemon re-reads config and re-indexes apps |
| `vida clear-history` | Sends `clear_history`; daemon truncates history table; prints "History cleared." |
| `vida ping` | Sends `ping`; prints "pong" on success, error message on failure |
| `vida status` | Prints daemon version, uptime, provider name, app count, history count |

### TR-02 · IPC Server

| Sub-ID | Requirement |
|--------|-------------|
| TR-02a | IPC server uses `net.Listen("unix", socketPath)` |
| TR-02b | Each accepted connection gets its own goroutine |
| TR-02c | Messages decoded with `json.NewDecoder` in a read loop |
| TR-02d | Daemon broadcasts `show`/`hide` to all connected UI clients |
| TR-02e | Socket file permissions: 0600 |

### TR-03 · AI Provider Interface

```go
package ai

import "context"

// Message is a single turn in a conversation.
type Message struct {
    Role    string // "user" | "assistant"
    Content string
}

// Token is a single streamed chunk from a provider.
type Token struct {
    Text  string
    Done  bool
    Error error
}

// AIProvider is the interface all AI backends must implement.
type AIProvider interface {
    // Name returns the provider identifier (e.g. "claude", "openai").
    Name() string
    // Query sends input to the provider and returns a channel of Token chunks.
    // The channel is closed when Done==true or Error!=nil is sent.
    // Cancelling ctx aborts the in-flight request.
    Query(ctx context.Context, input string, history []Message) (<-chan Token, error)
}
```

### TR-04 · Calculator Detection Heuristic

An input string is treated as a math expression if it satisfies **all** of the following:
1. Contains at least one digit
2. Contains at least one operator (`+`, `-`, `*`, `/`, `^`, `%`) OR a known function name (`sqrt`, `abs`, `floor`, `ceil`, `round`, `log`, `sin`, `cos`, `tan`)
3. Does not contain alphabetic sequences longer than the longest supported function name (10 chars) unless they are a known function name — this prevents "what is 2 times 3" from being detected as math

Implementation: use `go-expr` library or equivalent; attempt parse, if error → not a math expression.

### TR-05 · Fuzzy Search Algorithm

| Sub-ID | Requirement |
|--------|-------------|
| TR-05a | Use Smith-Waterman variant or prefix-weighted Levenshtein; acceptable alternative: `github.com/sahilm/fuzzy` |
| TR-05b | Score favors prefix matches and consecutive character matches over scattered matches |
| TR-05c | Search executes in < 10 ms for an index of 500 apps |
| TR-05d | Results with score == 0 are excluded |

### TR-06 · GTK4 UI Constraints

| Sub-ID | Requirement |
|--------|-------------|
| TR-06a | UI built with GTK4 and `gtk4-layer-shell` (CGo via `github.com/wmww/gtk4-layer-shell` Go bindings or direct cgo wrapper) |
| TR-06b | Alternative: `gioui.org` if GTK4 bindings prove unstable — decision to be made during Phase 1 prototype |
| TR-06c | UI renders at 60 fps minimum during streaming token display |
| TR-06d | No UI state is shared across goroutines without GTK main thread dispatch |
| TR-06e | UI binary links against system GTK4 and layer-shell shared libraries; not statically bundled |

### TR-07 · SQLite

| Sub-ID | Requirement |
|--------|-------------|
| TR-07a | Use `modernc.org/sqlite` (pure Go, no CGo) |
| TR-07b | Enable WAL mode: `PRAGMA journal_mode=WAL` |
| TR-07c | All DB access from a single goroutine via a serialised channel to avoid `SQLITE_BUSY` |
| TR-07d | Schema migrations use sequential integer versions stored in `PRAGMA user_version` |

## 6. Architecture Notes

### Startup Sequence
```
1. vida-daemon starts
   ├── Load config (or use defaults)
   ├── Open/migrate SQLite DB
   ├── Start IPC socket listener  ← daemon is "ready" here (< 200ms)
   └── Start app indexer goroutine (async, non-blocking)

2. vida-ui starts (launched by systemd user unit or alongside daemon)
   ├── Connect to IPC socket
   ├── Initialize GTK4 + layer-shell surface (hidden)
   └── Enter GTK main loop (waiting for "show" IPC)

3. Hyprland keybind fires
   └── vida show
       └── CLI connects to socket, sends {"type":"show"}, closes
           └── Daemon receives "show", broadcasts to connected UI clients
               └── UI receives "show", calls gtk_widget_show()
```

### Goroutine Model (daemon)
```
main goroutine
  └── IPC accept loop
        └── per-connection goroutine
              ├── read loop (decode JSON, dispatch to handler)
              └── write loop (encode JSON from result channel)

db goroutine (single, serialised all DB access)
app indexer goroutine (runs at startup + reload)
AI query goroutine (per query, cancellable via context)
```

### Key Packages
| Package | Purpose |
|---------|---------|
| `github.com/BurntSushi/toml` | TOML config parsing |
| `modernc.org/sqlite` | Pure-Go SQLite driver |
| `github.com/sahilm/fuzzy` | Fuzzy search for app launcher |
| `github.com/expr-lang/expr` | Math expression evaluation |
| GTK4 bindings (TBD) | UI rendering and layer-shell |
