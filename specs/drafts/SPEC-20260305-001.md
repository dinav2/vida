## 1. Purpose (WHY)

Linux power users on tiling WM setups (Hyprland/Wayland) lack a lightweight, AI-native command palette equivalent to Alfred or Raycast. Existing launchers are either too basic (rofi, wofi) or too heavy (Electron-based). **vida** is a keyboard-driven command palette that understands natural language, integrates with the OS at a native level, and stays out of the way until needed.

**Primary user**: A developer running Omarchy (Hyprland + Wayland) who wants sub-100ms access to apps, commands, calculations, and AI assistance through a unified interface.

---

## 2. Scope (WHAT)

### In Scope (MVP)
- Background daemon process with <30MB idle RAM footprint
- Wayland-native UI window using `wlr-layer-shell` protocol (no X11, no Electron)
- Global hotkey activation (configured via Hyprland keybind, not daemon-captured)
- App launcher with fuzzy search across installed applications
- Natural language processing via a pluggable AI provider interface
  - Claude (Anthropic) implementation
  - OpenAI implementation
  - Active provider swappable via config
- Inline calculator (math expressions evaluated locally, no AI call)
- Web search shortcuts (e.g., `g cats` вҶ’ open Google search)
- Unix socket IPC between daemon and UI process
- SQLite persistence for query history and basic preferences
- TOML-based configuration file (`~/.config/vida/config.toml`)
- CLI tool (`vida`) for management (reload config, clear history, etc.)

### Out of Scope (future specs)
- Clipboard history manager
- Emoji picker
- File finder / fuzzy file search
- Snippets and text expansion
- Plugin/skill extension system
- Hyprland window management integration (focus, move, resize)
- AI memory and personal context learning
- Themes and visual customization beyond defaults
- Packaging (AUR, flatpak, nix)

### Language & Runtime
**Go** вҖ” chosen for:
- Fast compile times and single binary distribution
- Goroutine model suits daemon + socket listener pattern
- Good CGo interop for GTK/layer-shell bindings
- Strong standard library for IPC, SQLite, HTTP

**Hotkey flow:**
1. User configures Hyprland keybind: `bind = SUPER, Space, exec, vida show`
2. `vida show` sends `SHOW` message to daemon via Unix socket
3. Daemon signals UI process to raise its layer-shell surface
4. UI is always-running but hidden (not respawned each time)

### AI Provider Interface

```go
type AIProvider interface {
    Name() string
    Query(ctx context.Context, input string, history []Message) (<-chan Token, error)
}

type Token struct {
    Text  string
    Done  bool
    Error error
}
```

Implementations:
- `ClaudeProvider` вҖ” uses `claude-sonnet-4-6` (or configurable model), Anthropic Messages API
- `OpenAIProvider` вҖ” uses `gpt-4o` (or configurable model), OpenAI Chat Completions API with streaming

### Query Router (priority order)
1. **Calculator** вҖ” regex match for math expression вҶ’ evaluate with `go-expr` or similar, instant result
2. **Web shortcuts** вҖ” prefix match (`g `, `gh `, `yt `, etc.) вҶ’ open browser with xdg-open
3. **App launcher** вҖ” fuzzy match against indexed `.desktop` files вҶ’ show ranked list
4. **AI fallback** вҖ” anything else вҶ’ stream to active AI provider

### Config (`~/.config/vida/config.toml`)
```toml
[ai]
provider = "claude"        # "claude" | "openai"
model = ""                 # empty = use provider default

[ai.claude]
api_key = ""               # or set ANTHROPIC_API_KEY env var
model = "claude-sonnet-4-6"

[ai.openai]
api_key = ""               # or set OPENAI_API_KEY env var
model = "gpt-4o"

[hotkey]
# vida is triggered by: bind = SUPER, Space, exec, vida show
# No daemon-side hotkey capture needed

[search]
[search.shortcuts]
g  = "https://www.google.com/search?q=%s"
gh = "https://github.com/search?q=%s"
yt = "https://www.youtube.com/results?search_query=%s"
dd = "https://duckduckgo.com/?q=%s"

[ui]
width = 640
position = "center"         # "center" | "top"
max_results = 8
```

### Data Storage
- SQLite at `~/.local/share/vida/vida.db`
- Tables: `queries (id, input, result_type, result_preview, created_at)`
- Accessed only by daemon; UI reads history via IPC

### IPC Protocol (Unix socket, newline-delimited JSON)
```
вҶ’ daemon:  {"type": "show"}
вҶ’ daemon:  {"type": "query", "input": "hello world"}
вҶҗ daemon:  {"type": "result", "kind": "ai_stream", "token": "Hello", "done": false}
вҶҗ daemon:  {"type": "result", "kind": "ai_stream", "token": "!", "done": true}
вҶҗ daemon:  {"type": "result", "kind": "app_list", "apps": [...]}
вҶҗ daemon:  {"type": "result", "kind": "calc", "value": "1764"}
```

## 5. Test Strategy

### Unit Tests
- Calculator: expression parsing and evaluation edge cases
- Query router: routing logic for each input type
- App indexer: parsing `.desktop` files, fuzzy ranking
- AI providers: mocked HTTP, verify request format, stream parsing
- Config: loading, defaults, env var override

### Integration Tests
- Daemon starts, socket is created, responds to `show`/`query` messages
- Claude provider: real API call (skipped in CI unless `ANTHROPIC_API_KEY` set)
- OpenAI provider: real API call (skipped in CI unless `OPENAI_API_KEY` set)
- SQLite: write query, read history, clear history

### Manual / E2E (documented, not automated in MVP)
- Hotkey triggers UI on Hyprland
- Layer-shell surface appears above all windows
- Escape dismisses without killing daemon
- App launch opens application

## 6. Implementation Plan

### Phase 1: Skeleton (daemon + IPC + UI shell)
1. Set up Go module (`github.com/dinav2/vida`)
2. `vida-daemon`: Unix socket server, basic message handling
3. `vida-ui`: GTK4 window with `gtk4-layer-shell`, IPC client, text input
4. `vida show` command wires hotkey вҶ’ socket вҶ’ UI reveal
5. Verify sub-100ms show latency

### Phase 2: App Launcher
1. Index `.desktop` files from standard paths
2. Fuzzy search with ranking
3. Display results in UI list
4. Enter to launch via `gio open` / `xdg-open`

### Phase 3: Calculator & Shortcuts
1. Math expression detection and evaluation (local, no AI)
2. Web shortcut prefix matching and browser open
3. Both appear inline as UI result types

### Phase 4: AI Integration
1. Define `AIProvider` interface
2. Implement `ClaudeProvider` with streaming
3. Implement `OpenAIProvider` with streaming
4. Config-driven provider selection
5. Stream tokens to UI as they arrive

### Phase 5: Persistence & Polish
1. SQLite setup, query history write
2. Up arrow history browsing in UI
3. `vida reload`, `vida clear-history` CLI commands
4. README with install + Hyprland config instructions

## 8. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| GTK4 layer-shell CGo complexity | Medium | High | Prototype early; fallback to `wlroots` direct or `gtk3-layer-shell` |
| Sub-100ms UI show latency | Medium | Medium | Keep UI process alive but hidden; measure at Phase 1 |
| AI streaming latency feels slow | Low | Medium | Show spinner immediately, stream tokens as received |
| CGo increases binary size / build complexity | Low | Low | Evaluate `gio` (pure Go, native Wayland) as alternative |
| API key management security | Low | High | Keys from env vars only in CI; config file with 600 perms |
