# PRD — vida Search Input Wiring (SPEC-20260307-002)

## Problem Statement

The vida UI currently shows a text field but it is disconnected — typing does nothing. Users need the palette to respond to keystrokes in real time: show calculator results instantly, expand shortcuts, list matching apps, and fall back to the AI provider for natural-language queries. The result must feel instantaneous for local handlers (calc, shortcuts, apps) and show incremental streaming for AI.

## Scope

Wire the GTK4 entry widget to the daemon IPC so that every keystroke triggers a query and the result is displayed below the input field. This spec covers the UI-side wiring and result rendering only; the daemon routing logic already exists (SPEC-20260305-001).

## User Stories

1. **Calculator** — User types `sqrt(144)`, sees `12` appear immediately below the input with no network call.
2. **Shortcut** — User types `g linux kernel`, sees a "Open in browser" button with the expanded URL.
3. **App launcher** — User types `fire`, sees a list of matching apps (Firefox, …); Enter or click launches the top result.
4. **AI query** — User types `explain what inode is`, sees tokens stream in as the AI responds.
5. **Empty state** — Input cleared → results area clears, no query sent.
6. **Cancel in-flight** — User clears input or hides window while AI is streaming → request cancelled, no leftover text.
7. **Enter to act** — Pressing Enter on a calc result copies it; on a shortcut opens the URL; on an app launches it; on an AI result copies the response.
8. **Escape** — Always hides the window and cancels any in-flight AI request.

## Requirements

### Functional
- FR-01: Entry widget sends a `query` IPC message on every keystroke (debounced 80 ms for AI, immediate for calc/shortcut/app).
- FR-02: Daemon `query` response `kind` field drives result rendering:
  - `calc` → single-line result string
  - `shortcut` → URL label + "Open" button
  - `app_list` → scrollable list of app names/icons
  - `ai_stream` → streaming text area (tokens arrive as separate `result` messages)
  - `empty` → clear results area
- FR-03: App launch executes the app's `Exec` field (via `g_app_info_launch` or `gio`).
- FR-04: Enter key acts on the current top result (copy calc, open shortcut URL, launch top app).
- FR-05: Window show (`vida show`) resets the input field and clears results.
- FR-06: AI streaming is cancellable — closing/hiding the window sends a `cancel` IPC message with the query ID.
- FR-07: Debounce: calc/shortcut/app results appear within 50 ms of last keystroke; AI query is sent after 80 ms idle.

### Technical
- TR-01: All GTK widget manipulation happens on the GLib main thread (use `g_idle_add` for callbacks from IPC goroutine).
- TR-02: IPC query uses the existing one-shot `ipc.Conn` (no persistent connection needed for queries).
- TR-03: AI streaming requires a persistent connection; reuse `ipc.PersistentConn` pattern.
- TR-04: Result rendering is purely additive — a new query replaces the previous result widget entirely.
- TR-05: The daemon must support a `cancel` message that aborts an in-flight AI stream by query ID.
- TR-06: App launch must strip `%u`/`%f`/`%U`/`%F` placeholders from the Exec field (already done in `apps.ExpandExec`).

## Out of Scope

- History/recency ranking of app results
- Multi-monitor output selection
- Custom result renderers / plugins
- Keyboard navigation within the app list (arrow keys) — deferred to SPEC-MVP-03
- AI conversation history — deferred

## Success Criteria

- AC-01: Typing `2+2` shows `4` within 50 ms.
- AC-02: Typing `g hello` shows the expanded Google URL.
- AC-03: Typing `firefox` shows Firefox in the app list; Enter launches it.
- AC-04: Typing a sentence triggers AI streaming; tokens appear progressively.
- AC-05: Clearing input clears results with no crash.
- AC-06: Hiding window while AI streams cancels the request (daemon logs confirm).
- AC-07: `go test ./...` passes with 0 failures after implementation.
